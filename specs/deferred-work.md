# Deferred Work

Real-but-not-now findings from code review, kept here so they aren't lost or silently re-litigated in a later story.

## From Story 1-1 (repo-scaffold-and-css-tokens) review

1. **`--surf1` has no `[data-theme="dark"]` override** in `public/css/theme.css`. Inherited faithfully from `../TM Suite/public/css/theme.css`, which has the same gap upstream — this is not a porting error. Currently inert (`--surf1` is unused in `base.css`). Fix here (and ideally upstream in TM Suite too) the first time a story actually consumes `var(--surf1)` in a dark-mode-visible element.
2. **`express.static` is registered before the `/` route**, so a future `public/index.html` would silently shadow the `homePage()` handler with no error. Latent only — no such file exists yet. Worth a comment or an ordering fix the first time a story adds static HTML under `public/`.
3. **No 404 handler or centralised error-handling middleware.** Acceptable for a one-route skeleton; add when routes with real async I/O (auth, dossier views) land in stories 1-3/2-1, so an unhandled rejection doesn't fall through to Express's default error page.
4. **Google Fonts CDN has no local fallback, SRI, or CSP.** Low severity for a placeholder page; revisit as part of a production-hardening pass (also consider `helmet` for basic security headers) before this is player-facing for real.
5. **The `isMain` check (`process.argv[1] === fileURLToPath(import.meta.url)`) is fragile** across symlinked or extensionless invocations (the `es-main` idiom exists precisely for this). Fine for the documented `npm start` / `node server/index.js` invocation; revisit if the deploy target ever launches the process differently.
6. **`PORT=0` produces a misleading startup log** (`http://localhost:0`) since Node binds a random ephemeral port in that case but the log doesn't reflect it. Cosmetic; not worth a fix unless something starts scripting against the logged URL.

## From Story 1-2 (mongo-snapshot-script) review

1. **BLOCKING FOR STORY 2-1 — `data/snapshot.json` carries every character's full document and every `character_dossier` fact (including `st_hidden: true` secrets) in the clear.** This is correct and intentional for this story — architecture.md places redaction at the Express serving layer, not the snapshot — but it means the committed file has no protection of its own. Story 2-1's implementation MUST: (a) implement the owner-full vs whitelist-summary projection server-side, never trusting the client; (b) never `express.static()` (or otherwise directly serve) the `data/` directory or `snapshot.json` itself; (c) be reviewed with this specifically in mind — a single misconfigured static route would leak every character's secrets and every player's `st_hidden` facts at once. Do not let story 2-1's code review treat this as optional.
2. **`toPlain()` only special-cases `ObjectId` and `Date`.** Other BSON types (`Decimal128`, `Long`, `Binary`, `Timestamp`) would serialize as their internal object structure rather than a sensible plain value — no crash, silent data corruption. None of the four collections currently store these types (verified against live data), so this is dormant risk, not an active bug. Revisit if a future collection or field introduces a non-string numeric/binary BSON type.
3. **The AC #3 "no writes" test is a lexical source-scan, not a runtime guarantee.** It was hardened this story to also catch `$out`/`$merge`/`db.command`/`runCommand`/bulk-op-initializers, but a lexical scan can still be evaded by sufficiently indirect code (e.g. a dynamically-constructed method name). The only true enforcement is the Atlas database user's read-only IAM role, which is a manual Atlas-console step Angelus performs outside this codebase (see README). Treat the test as a tripwire against accidental writes, not as the actual security boundary.

## From Story 1-3 (discord-oauth-reuse) review

1. **In-memory token cache (`server/middleware/auth.js`) grows unbounded** — expired entries are never swept, only overwritten or deleted on an explicit 401. Not a confidentiality issue (cached identity is only ever the identity Discord itself returned for that exact token), just unbounded memory growth over the process lifetime. Negligible at this app's real scale (dozens of players); revisit only if that assumption ever changes.
2. **`snapshot-store.js` accessors (`getPlayers`, `getCharacters`, etc.) return live references into the in-memory snapshot, not copies.** `buildUserFromPlayer` already defends itself by copying `character_ids`, so there is no present bug. But any future route that mutates a returned character/player object in place would silently corrupt the shared boot snapshot for every subsequent request. Worth a shallow-copy or `Object.freeze` pass on the accessors before story 2-1/2-2 start reading `getCharacters()`/`getDossiers()` directly, so this can't become a footgun by omission.
3. **Callback route network-failure handling was ported without the middleware's try/catch, then patched here** — now consistent (both wrap their Discord `fetch` calls). Noting only so a future refactor that touches one doesn't quietly drop the other out of parity again.
