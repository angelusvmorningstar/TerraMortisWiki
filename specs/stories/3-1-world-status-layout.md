# Story 3.1: world-status-layout

Status: done

> **Frontend-only presentation rework, built to a design-locked mockup.** This story does NOT touch the server: `GET /api/world` and `GET /api/characters` already return everything the new layout needs, and neither route (nor any schema) changes. The whole of this story lives in `public/court.html`, `public/js/world/court.js`, and `public/css/components.css`. The design is not open for re-litigation: an approved static mockup at `specs/mockups/3-1-world-status-layout/mockup.html` is the source of truth, and its own `.mockup-note` block records the locked decisions verbatim (no player-name field, clan as text not icon, Regencies = Regents only). Those decisions are folded into the ACs and Dev Notes below as authoritative. The projection posture of Story 2-2 carries forward unchanged and needs no new gating here: both endpoints hand this page pre-projected, allowlist-safe payloads (`/api/world` holders are built from `HOLDER_FIELDS`; `/api/characters` entries from `SUMMARY_FIELDS`), so the client only ever reads fields the server already chose to expose. This story adds no new field to any wire response.

## Story

As a player,
I want the Court page to present the court, the regencies, and the full active roster as three collapsible sections, with each person's covenant, clan, and any office they hold visible at a glance,
so that I can find who holds power and who belongs to which faction without scrolling through flat, undifferentiated lists.

## Acceptance Criteria

1. **Frontend-only, no backend change.** No server file changes: `server/routes/world.js`, `server/routes/characters.js`, `server/mongo-store.js`, and every schema file stay byte-identical. No new endpoint is added and no existing endpoint's response shape changes. The entire deliverable is `public/court.html`, `public/js/world/court.js`, and tokens-only additions to `public/css/components.css`. `GET /api/world` (returning `{ territories: [{ territory, regent, lieutenant }], titleGroups: [{ honorific, holders }] }`) and `GET /api/characters` (returning `{ characters: [...] }`, the active non-retired roster in the `SUMMARY_FIELDS` shape: `_id`, `name`, `honorific`, `moniker`, `clan`, `covenant`, `bloodline`, `apparent_age`, `retired`) are consumed exactly as they already exist.

2. **Three accordion sections, native `<details>`/`<summary>`, fixed order.** `public/js/world/court.js` replaces its current two flat sections (Territories cards + Court-and-titles grid) with THREE collapsible sections rendered using native `<details>` / `<summary>` elements, in this exact order: **Court**, then **Regencies**, then **Who's Who**. The collapse/expand interaction is provided entirely by the native element (no JS click-handler, no library) so keyboard operation and the disclosure semantics come for free. Each section header (`<summary>`) shows the section label, a rotating chevron affordance, and a count badge reflecting the number of rows the section actually renders. All three default to open, matching the mockup.

3. **Court section: one row per office-title holder.** The Court section lists every character who holds a court office title, one row each, sourced from `/api/world` `titleGroups` (flattened: because `honorific` is a single field per character, each holder appears in exactly one group and therefore exactly once). Each row shows, in the mockup's layout: the holder's display name (via ported `displayName()`), the office title as a red pill badge (the `--crim` "roster-badge" pill), the holder's clan as plain text, and the covenant crest icon. The row links to that character's profile page (`/character.html?id=<_id>`). Clan and covenant are NOT present on the `/api/world` holder object (it carries only `_id`/`name`/`honorific`/`moniker`), so they are resolved by cross-referencing the holder `_id` against the `/api/characters` roster (AC #6). A holder whose `_id` is absent from the active roster still renders (name + title badge), with clan text and covenant icon simply omitted (honest gap, never a guess).

4. **Regencies section: one row per territory, Regent only.** The Regencies section lists one row per territory from `/api/world` `territories`, showing that territory's **Regent only**. The territory name is the row's badge; the row's name slot is the Regent's linked display name, followed by the Regent's clan text and covenant icon (resolved by cross-referencing the regent holder's `_id` against the roster, AC #6). **Lieutenants never appear in this section** (locked mockup decision: a Lieutenant is a territorial appointment, not a personal title, so it carries no personal badge here). A territory whose Regent seat is vacant (the `/api/world` `regent` field is `null`, already meaning null/absent/unmatched/retired per Story 2-2's server assembly) renders an honest **Vacant** in the name slot with the territory-name badge still shown, never dropped and never a fabricated holder.

5. **Who's Who section: full active roster grouped by covenant.** The Who's Who section renders the full active (non-retired) roster from `/api/characters`, grouped into sub-sections by covenant. Each covenant sub-group is introduced by its own heading bar carrying the covenant name plus the covenant crest icon (the mockup's `roster-group__head`; NOT a nested `<details>` accordion). Each character row shows the display name (linked to the profile page), the clan as plain text, and - **only if that character also holds an office title** - that office title as a red pill badge, cross-referenced against the SAME title-holder data (`/api/world` `titleGroups`, joined by `_id`) that drives the Court section, so the two sections can never disagree about who holds what. A character whose covenant is empty/absent or unrecognised falls into an honest catch-all group (e.g. "No covenant recorded") rather than a broken or guessed icon.

6. **Cross-reference join is pure frontend assembly, `_id`-keyed, both sides String-normalised.** "Which characters hold a title" and "what are this holder's clan/covenant" are derived client-side by joining `/api/world`'s `titleGroups` holders and `/api/characters`'s roster on `String(_id)` (both sides `String(...)`-normalised, mirroring the repo's existing join discipline in `server/routes/world.js` / `../TM Suite/public/js/data/helpers.js`). No new server logic and no new endpoint: both payloads are already allowlist-projected by their existing routes, so this assembly reads only fields the server already chose to expose and introduces NO new field on any wire response. There is nothing new to gate here (the [PROJECTION] boundary lives in the two server routes and is untouched); the story simply must not fabricate a field, must not invent a holder, and must handle an `_id` present in one payload but absent from the other as an honest gap.

7. **Covenant icons reuse the existing ported asset + slug map; no re-port, no duplication.** Covenant crests use the already-ported `.char-card__icon--<slug>` CSS mask pattern (`public/css/components.css`) and the covenant-name to icon-slug map already defined as `COVENANT_ICON_SLUG` in `public/js/characters/list.js` (the five canonical covenants: Carthian Movement, Circle of the Crone, Invictus, Lancea et Sanctum, Ordo Dracul; SVGs already present in `public/img/covenant/`). Do NOT re-port any icon and do NOT hand-duplicate the map: share it (extract to a small shared module both `list.js` and `court.js` import) or mirror it exactly with a comment pointing at the canonical copy. An unrecognised/absent covenant falls back honestly (letter-monogram or a plain "No covenant recorded" group), never a broken icon.

8. **Locked design decisions (from the approved mockup `.mockup-note`), authoritative.** These are settled; do not re-open them:
   - (a) **No player-name / real-name line** under any character. The wiki's data model has no such field at all, so there is nothing to redact: it is simply never fetched. (TM Suite's internal admin view shows ST-only real names; this app does not.)
   - (b) **Clan is a plain text label everywhere on this page, never an icon.** Only the five covenant SVGs exist as web assets in this repo; porting the matching clan icons is a parked follow-up (`specs/ux-vision-log.md`), explicitly out of scope for this story.
   - (c) **Regencies lists Regents only** (see AC #4): Lieutenants are a territorial role, not a personal title, and carry no badge in this layout.

9. **CSS folded from the mockup, tokens-only, no bare literals.** The mockup's inline `<style>` block is folded into `public/css/components.css` (reviewed and adapted, not blindly pasted). The genuinely-new classes are tokens-only: `.roster-section` (+ `__head`, `__chevron`, `__count`, `__body`), `.roster-group__head` (+ `__icon`), `.roster-row` (+ `__name`, `__badges`), `.roster-badge`, `.roster-clan`, `.roster-cov-icon`. No bare hex, no `rgba()` literal, and no inline `style="..."` anywhere in the markup or in JS-rendered HTML. In particular: (i) the mockup's `var(--txt-on-dark, #fff)` on `.roster-badge` MUST become a bare `var(--txt-on-dark)` (the token exists in both the light and dark theme blocks of `theme.css`; the `#fff` literal fallback is a bare-hex violation and must not ship); (ii) the mockup rendered covenant/group icons with an inline `style="mask-image:url(...)"` as a static-HTML convenience, which is a mockup-only shortcut: production JS-rendered HTML MUST select the crest via the `.char-card__icon--<slug>` modifier classes (or dedicated `.roster-cov-icon--<slug>` / `.roster-group__icon--<slug>` modifier classes added to `components.css`), never an inline `style` attribute. Reuse `.site-nav`, `.content`, `.content__header`, and the existing tokens (`--gold2`, `--accent`, `--crim`, `--surf`, `--surf2`, `--bdr`, `--bdr2`, `--bdr3`, `--txt2`, `--txt3`, `--txt-on-dark`, `--fh`, `--fl`, `--ft`).

10. **The mask-icon `display: inline-block` gotcha MUST be preserved.** Every mask-image icon `<span>` on this page (`.roster-group__icon` and `.roster-cov-icon`, and any modifier applied to them) MUST carry `display: inline-block` (or `block`/`flex`). A `mask-image` applied to a default-`inline` span collapses to zero width/height and renders nothing at all: this exact bug was caught and fixed during the mockup render, and the mockup's folded CSS already has the fix on both icon classes. Do not drop `display: inline-block` when folding the CSS in, and do not reintroduce a plain-`inline` icon span. (The existing `.char-card__icon` is a flex child with an explicit box, which is why it renders; a bare `<span>` crest without an explicit `display` will silently vanish.)

11. **Honest empty/vacant states; counts are truthful; retired never appears.** Each section renders an honest plain state when empty: no title-holders ("No titled offices recorded"), no territories ("No territories recorded"), an empty roster ("No characters recorded"), never a placeholder implying a bug. A vacant regency renders "Vacant" (AC #4). Retired characters do not appear anywhere on this page: the `/api/characters` endpoint already excludes them and `/api/world` already resolves a retired regent to a vacant seat, so no client-side retired handling is added or needed (and none of the mockup's rows imply otherwise). Each section's count badge reflects the number of rows actually rendered.

12. **Auth, loading, and error posture unchanged.** Both fetches (`/api/world` and `/api/characters`) go through the shared `apiGet` helper (`public/js/data/api.js`: Bearer token, redirect to `login.html` on a missing token or 401/403). A fetch/store error surfaces the existing `.hero__status--error` status line rather than a blank page (matching the current `court.js`). The page inherits the whole-site login gate (Story 1-3); this story adds NO per-viewer projection logic (that is Story 3-2's concern and is out of scope, AC #14).

13. **British English throughout; no em-dashes in any user-facing string, including the `<title>`.** The page `<title>` stays `Terra Mortis Wiki: Court` (colon, no em-dash, per the Story 2-2 F1 precedent). Section labels are "Court", "Regencies", "Who's Who". All rendered copy uses British spelling (e.g. Colour, Honour, Favour, capitalise where such words occur). Code comments may keep the repo's em-dash style; user-facing strings may not.

14. **Explicitly out of scope (named so this story does not absorb it).** Do NOT build, stub, or partially wire any of the following:
   - The City/Covenant/Clan **status ladder** feature: that is **Story 3-2**, separately scoped, and it involves genuinely new per-viewer access-control/projection logic (a non-ST viewer must see only the Covenant/Clan status ladder for the covenant(s)/clan their OWN character belongs to, mirroring TM Suite's `covenantListFor` / `clanRowsFor` gating exactly). No status-ladder UI, data, or endpoint appears in this story.
   - **Clan icons as SVG assets**: clan stays plain text this story (AC #8b); the clan-icon port is parked in `specs/ux-vision-log.md`.
   - Any change to `/api/world` or `/api/characters`, or to the World **map** page (`public/world.html` / `public/js/world/map.js`), which is untouched.

### The mask-icon `display: inline-block` gotcha (AC #10) - do not reintroduce

A CSS `mask-image` recolours a silhouette by masking a background-colour fill, but it needs a box to fill. A `<span>` defaults to `display: inline`, which has no intrinsic width/height, so `background-color` + `mask-image` on a bare inline span collapses to a zero-size box and the icon renders as **nothing at all** - no error, no broken-image glyph, just an invisible span. This was hit and fixed during the mockup render: both `.roster-group__icon` and `.roster-cov-icon` in the mockup carry `display: inline-block` for exactly this reason. When folding the CSS into `components.css`, keep `display: inline-block` (or `block`/`flex`) on every mask-icon span. The existing `.char-card__icon` does not need it only because it is a flex child with an explicit `width`/`height` box; a fresh crest span on this page is not, so it must set `display` itself. Verify each icon has a non-zero rendered box in-browser before signing off.

### Two mockup-only shortcuts to correct when folding in (AC #9)

1. **`var(--txt-on-dark, #fff)` -> `var(--txt-on-dark)`.** The `#fff` literal fallback is a bare-hex violation of the normalised-CSS rule. The token exists in both themes, so drop the fallback.
2. **Inline `style="mask-image:url(...)"` on the covenant/group icons is a static-HTML convenience only.** Production JS-rendered HTML must not carry inline `style` attributes (normalised-CSS rule). Select the crest through modifier classes (`.char-card__icon--<slug>`, or add `.roster-cov-icon--<slug>` / `.roster-group__icon--<slug>` to `components.css`) keyed off `COVENANT_ICON_SLUG`.

### F1 (PATCHED in-scope): the catch-all-last test was non-discriminating

`buildWhosWho` sorts the covenant groups alphabetically but forces the "No covenant recorded" catch-all last via an explicit special-case. The unit test `Who's Who: empty/absent/unrecognised covenant falls into an honest catch-all, last` used a fixture whose ONLY recognised covenant was "Invictus". "No covenant recorded" sorts AFTER "Invictus" alphabetically (`'No covenant recorded'.localeCompare('Invictus') === 1`), so the catch-all landed last by coincidence of the label, not by the rule. Removing the special-case left the test green: it did not protect the behaviour it claimed to.

This matters in live data. "Ordo Dracul" is one of the five canonical covenants and sorts AFTER the catch-all label (`'No covenant recorded'.localeCompare('Ordo Dracul') === -1`). Without the special-case, a roster containing Ordo Dracul characters (the live data does) would render the "No covenant recorded" group BEFORE the "Ordo Dracul" group, violating AC #5's honest-catch-all-last intent, and no test would catch it. The production CODE was correct; the TEST was weak.

Patch (test-only, in-scope `court-view.test.js`): added an "Ordo Dracul" character to the fixture and asserted `names.indexOf('Ordo Dracul') < names.indexOf('No covenant recorded')`. Discrimination proven: with the special-case the strengthened test passes (9/9); reverting the special-case to a plain `localeCompare` now FAILS it (8/9); restored to green. No production code changed; the test count stays 108 (the ninth court-view case gained an assertion, it was not a new case).
